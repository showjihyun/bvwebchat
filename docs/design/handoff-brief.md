# 디자인 Hand-off 브리프 — 클로드 디자인용

> **목적**: 확정된 요구사항(RQ v1)과 아키텍처 결정(ADR)을 클로드 디자인이
> 바로 쓸 수 있는 형태로 정리한다. 인터뷰 H 결정(질문 26)에 따라 디자인은
> **클로드 디자인 선행 → 산출물(DESIGN.md)을 UI 구현이 이어받는** 방식이다.
>
> **이 문서의 위치**: 입력(브리프)이지 진실 공급원이 아니다. 진실 공급원은
> 산출물인 `docs/design/DESIGN.md`가 된다 (인터뷰 질문 27 결정).
>
> - 근거: `specs/requirements.md` v1, `docs/adr/`, `specs/interview/answers.md` H섹션,
>   `specs/interview/design-answers.md`(디자인 인터뷰 — §3의 입력)
> - 절차: §4 참조. 산출물 수령 후 할 일이 여기 적혀 있다.
> - **§2 = 확정 요구사항**(스펙·ADR 근거), **§3 = 사용자가 결정한 디자인 방향**.
>   §3은 더 나은 안이 있으면 이유와 함께 제안할 수 있다. §2는 스펙 개정 없이 못 바꾼다.

## 1. 제품 요약

사내망에서 쓰는 실시간 웹 채팅. room 단위 대화와 전체 공지용 global 채널을
제공한다. 계정 없이 닉네임만으로 입장한다.

| 항목 | 값 | 근거 |
|---|---|---|
| 배포 환경 | 사내망/로컬 단일 서버 | RQ-17 |
| 주 사용 뷰포트 | 데스크톱 브라우저 위주 (제안) | **디자인 재량** — 스펙 근거 없음. DESIGN.md가 정한다 |
| 동시 사용자 | 최대 100명 | RQ-16 |
| 수명 | L&L 산출물 + 발전 베이스 | answers E16 |
| 마감 | 7/31 | RQ-05 |
| 구현 스택 | React + TypeScript + Vite | ADR-0001 |

## 2. 화면과 요구사항 (디자인이 반영해야 할 확정 사항)

### 2-1. 입장 화면

- 닉네임 입력 **하나**로 시작. 계정·비밀번호·회원가입 없음. (RQ-10)
- 이미 사용 중인 닉네임이면 시스템이 **자동 접미사**를 붙여 고유화한다
  (예: `alice` → `alice-2`). 거부가 아니라 자동 변형이므로, 사용자에게
  "부여된 이름"을 알려주는 표현이 필요하다. (RQ-10)
- 같은 브라우저에서 새로고침하면 동일 사용자로 복원된다 — 재입력 화면이
  다시 뜨지 않는다. (RQ-10, ADR-0003)

### 2-2. 메인 화면 (3단 구성 — 디자인 가정, 조정 가능)

> 스펙은 아래 3개 영역이 **표시될 것**만 요구한다. "3단" 배치는 제안이며
> 디자인 재량이다.

| 영역 | 내용 | 근거 |
|---|---|---|
| room 목록 | 전체 공개 — 모든 room이 보인다. **global이 항상 존재하며 목록에 표시**된다(삭제 불가·예약 이름). room 생성 진입점 포함 | RQ-13, ADR-0004 |
| 채팅 영역 | 입장 시 **최근 50개** 히스토리 + 실시간 수신. 메시지 입력창 | RQ-11, RQ-02 |
| 참여자 목록 | 현재 room의 참여자. **온라인/오프라인 표시, 타이핑 표시는 스코프 제외** | RQ-15 |

- **global 메시지는 room에 있는 동안에도 수신**된다. (RQ-04, 인터뷰 질문 13)
  > 단, **전달과 표시는 다르다**: 사용자는 global에 자동 참여하므로 항상
  > 수신하지만(RQ-04 충족), 화면에 어떻게 드러낼지는 디자인 결정이다.
  > 이 앱은 §3-3 결정에 따라 "room 목록의 global 항목에 안 읽음 표시"만 한다.
- room 메시지는 **참여자에게만** 전달된다 — 다른 room 대화가 섞여 보이면 안 된다.
  (RQ-02)

### 2-3. room 생성

- 누구나 생성 가능. 권한 제한 없음. (RQ-12)
- **이름 중복 불허** — 이미 있는 이름이면 거부 피드백. (RQ-13)
- `global`(대소문자 무관)은 **예약 이름**이라 생성 거부. (ADR-0004)
- 마지막 참여자가 나가면 room이 자동 소멸한다 → 목록에서 사라지는 상황이
  정상 동작이다. (RQ-12)

### 2-4. 상태·피드백

- 재연결 중(새로고침·일시 단절, 30초 유예) 상태 표현이 필요할 수 있다. (ADR-0003)
- 에러 표현이 필요한 지점: room 이름 중복(RQ-13), 예약 이름(ADR-0004)은 스펙 근거.
  닉네임 미입력·전송 실패는 **디자인 추론**(스펙 근거 없음).
- 서버 재시작 시 메시지가 사라지는 것은 **정상 동작**이다 — 영속 저장 없음.
  (RQ-11, ADR-0002)

## 3. 디자인 방향 (확정 — 디자인 인터뷰 2026-07-17)

> 근거: `specs/interview/design-answers.md`. 아래는 **사용자가 결정한
> 방향**이므로 따르되, 구체적 값(정확한 색·수치·비율)은 디자이너가 정한다.
> 이 결정들과 다른 방향이 더 낫다고 판단되면 **이유와 함께 제안**할 것.

### 3-1. 정체성

| 항목 | 방향 |
|---|---|
| 참조점 | **Slack** — 좌측 네비게이션 + 플랫한 메시지 리스트, 높은 정보 밀도 |
| 톤 | **절제된 업무 도구** — 대화 내용이 주인공. UI는 배경으로 물러난다. 화려함·장식 배제 |
| 밀도 | **컴팩트** — Slack 수준. 한 화면에 많은 메시지. 스크롤 빈도를 줄이는 게 목표 |
| 브랜드 | **제약 없음** — 팔레트·네이밍 자유. 사내 컬러·로고 없음 |

### 3-2. 시각 언어

| 항목 | 방향 |
|---|---|
| 컬러 | **뉴트럴 + 액센트 1개** — 회색조 기반, 강조색 하나만(선택·활성·중요 표시에 한정) |
| 모드 | **라이트만** (v1) — 다크는 만들지 않는다. 단, 토큰 구조는 나중에 다크를 얹을 수 있게 |
| 타이포 | **시스템 폰트** — Pretendard/맑은 고딕 등 OS 기본 스택. 웹폰트 금지(사내망에서 외부 CDN 접근이 막힐 수 있다). **한국어 UI**이므로 한글 줄높이·자간을 명시할 것 |
| 모션 | **최소한** — 상태 전환·호버 정도. 새 메시지 등장에 화려한 애니메이션 금지(업무 중 방해) |
| 접근성 | **WCAG AA** — 본문 대비 4.5:1 이상, 키보드 포커스 링 필수. 팔레트 설계 시점에 반영할 것 |

### 3-3. 컴포넌트 결정

| 컴포넌트 | 결정 |
|---|---|
| 메시지 | **플랫 리스트** — 말풍선 없음. 이름 + 시간 + 본문 한 줄 흐름. 본인/타인 구분은 정렬이 아닌 다른 방식으로 |
| 아바타 | **이니셜 아바타** — 닉네임 첫 글자 + 닉네임 기반 자동 색상 원형. 이미지 업로드 없음 |
| room 목록 | 안 읽음은 **이름 굵게 + 작은 점**. 숫자 배지 금지(개수를 세지 않는다 — 스펙에 없음). global은 상단 고정 |
| global 알림 | room 대화 중 global 메시지 도착 시 **room 목록의 global 항목에만 안 읽음 표시**. 현재 room 대화 흐름에 끼워넣거나 배너로 띄우지 않는다 (전달과 표시의 분리 — §2-2 주 참조) |
| 참여자 패널 | **우측 상시 노출**. 토글로 숨기지 않는다 |
| 입장 화면 | **최소 폼 하나** — 빈 화면 중앙에 닉네임 입력창 + 버튼. 랜딩 페이지·마케팅 문구 없음 |
| 입력창 | 기본/포커스/전송 불가 상태 |
| 에러 | §2-4 목록 |

### 3-4. 산출물 형식 (DESIGN.md)

1. **디자인 원칙** — 위 방향을 원칙 문장으로 정리
2. **컬러 팔레트** — **시맨틱 토큰**으로 (예: `--color-bg-surface`, `--color-text-muted`,
   `--color-accent`). React+TS 구현으로 그대로 옮길 것이므로 토큰 명명이 중요하다.
   각 토큰의 대비율을 병기할 것 (AA 검증 가능하게)
3. **타이포 스케일** — 시스템 폰트 스택 + 한글 기준 줄높이·자간
4. **레이아웃** — 3개 영역(room 목록/채팅/참여자)의 배치·비율. 뷰포트 우선순위는
   스펙이 정하지 않았으므로 **여기서 결정해 명시**할 것
5. **컴포넌트 명세** — §3-3의 각 컴포넌트에 대해 상태별(기본/호버/선택/에러) 정의
6. **간격 체계** — 스케일 토큰 (컴팩트 밀도 기준)

### 스코프 밖 (디자인하지 말 것)

파일·이미지 전송, DM(1:1), 알림 푸시, 모바일 앱 전용 UI, 로그인/회원가입 화면,
온라인·타이핑 표시, 비공개/초대제 room. — 전부 v1 명시 제외(requirements §3).
필요해지면 스펙 개정이 먼저다.

디자인 인터뷰로 추가 제외된 것: **다크 모드**(D6), **안 읽음 숫자 배지**(D13 —
개수를 세는 것은 스펙에 없다), **프로필 이미지 업로드**(D8), **랜딩 페이지**(D15).

## 4. Hand-off 절차

### Step 1 — 클로드 디자인에 요청 (사용자)

이 문서의 §1~§3을 클로드 디자인에 전달한다. 붙여넣기용 프롬프트:

```
사내 웹 채팅 앱의 Design Style 가이드를 만들어줘. 산출물은 DESIGN.md 형식의
마크다운 하나 (§3-4의 6개 항목 구성).

아래 브리프에서:
- §2는 확정 요구사항 — 반드시 반영
- §3은 사용자가 결정한 디자인 방향 — 따르되, 더 나은 안이 있으면 이유와 함께 제안
- §3 "스코프 밖" 항목은 만들지 마

토큰은 시맨틱 명명으로, React+TypeScript 구현에 그대로 옮길 수 있게 해줘.

[docs/design/handoff-brief.md의 §1~§3 붙여넣기]
```

### Step 2 — 산출물 수령 (사용자 → Claude Code)

산출물 마크다운을 채팅에 붙여넣거나 파일 경로를 알려준다.

### Step 3 — 커밋·등재 (Claude Code)

1. `docs/design/DESIGN.md`로 저장, 브랜치 `chore/design-md` 생성
2. **CLAUDE.md 진실 공급원 목록에 4번으로 등재** (인터뷰 질문 27 결정).
   requirements·ADR·CLAUDE.md 아래에 둔다 — 디자인은 기능을 추가할 수 없으므로
   충돌 시 requirements가 이긴다. ADR과는 관할이 직교한다(디자인 스타일은
   ADR이 아니라 DESIGN.md 소관 — `docs/adr/README.md` 각주).
   등재 후 UI 구현이 DESIGN.md와 모순되면 드리프트로 취급된다.
3. CLAUDE.md 참조 맵 "UI 작업" 행에서 `(미확보 시 ... 절차)` 괄호를 제거해
   `docs/design/DESIGN.md`만 남긴다 — 브리프 포인터가 빠지면서 이 문서는 은퇴한다.
4. `docs/progress.md` Phase 3.5 체크, `docs/harness/changelog.md` 기록
5. **이 브리프는 hand-off 시점의 입력 기록으로 동결한다** — 이후 개정하지 않는다.
   스펙 변경은 requirements.md, 디자인 변경은 DESIGN.md에서 한다. 브리프와
   DESIGN.md가 어긋나면 **DESIGN.md가 이긴다** (브리프는 갱신 대상이 아니다).
6. PR 생성 → CI → `review-gate`(reviewer APPROVE) → 머지

### Step 4 — 이후 규칙 (인터뷰 질문 28 결정)

- 모든 UI 작업은 DESIGN.md를 이어받는다. 구현이 DESIGN.md와 어긋나면 드리프트다.
- 구현 중 디자인 변경이 필요하면 **DESIGN.md 개정을 코드 변경과 같은 PR에**
  담는다 (스펙 변경 규칙과 동일).
- 디자인은 ADR이 아니라 DESIGN.md가 진실 공급원이다 (`docs/adr/README.md` 각주).

### 검증 체크리스트 (Step 3 수행 시)

- [ ] 스코프 밖 항목(로그인 화면·DM·타이핑 표시·다크 모드·숫자 배지)이 섞이지 않았는가
- [ ] §3 결정과 어긋난 부분이 있다면 **이유가 제시**됐는가 (제안은 환영, 무단 이탈은 아님)
- [ ] 컬러가 뉴트럴 + 액센트 1개 구조인가, **대비율이 병기**됐는가 (WCAG AA)
- [ ] 폰트가 시스템 스택인가 (웹폰트·CDN 의존 없음 — 사내망 제약)
- [ ] 한글 기준 줄높이·자간이 명시됐는가
- [ ] 토큰이 시맨틱 명명인가 (React+TS 전환 가능성)
- [ ] 메시지가 플랫 리스트인가 (버블 아님), 아바타가 이니셜 방식인가
