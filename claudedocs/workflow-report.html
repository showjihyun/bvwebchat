bvwebchat — AI 에이전트 워크플로우 적용 보고서

room 참여 채팅 + global 채팅 웹앱을 소재로, 스펙 → ADR → TDD 파이프라인 → 독립 리뷰 → CI/CD로 이어지는 AI 에이전트 개발 워크플로우(하네스)를 설계·운영한 결과를 정리한다. 마감 7/31 대비 7/21에 전 요구사항 구현을 완료했다.

무엇을 만들었나

  • 제품: room 참여형 채팅 + 전체 공지용 global 채널. 닉네임 식별(계정 없음), 최근 50개 히스토리, room 자유 생성·자동 삭제, 참여자 목록, 안 읽음 배지, 새로고침 세션 복원.
  • 스택: Node.js + TypeScript, Socket.IO, React + Vite, Vitest. Docker 단일 컨테이너 배포 (사내망 단일 서버 대상).
  • 진짜 목적: 제품 자체보다, 1인 개발 환경에서 AI 에이전트에게 구현을 맡길 때 품질을 어떻게 보장하는가에 대한 워크플로우 실험. 사내 L&L(Lunch & Learn) 공유 겸용.
이 문서를 읽는 법
3~3-7장이 "무엇을 적용했나", 4~5장이 "그래서 무엇이 좋았나(증거 포함)", 6~8장이 "남은 것과 팀에 가져갈 것"이다. 모든 주장에는 저장소 내 파일·PR·테스트 출력 근거를 병기했다.

2. 워크플로우 전체 구조

핵심 아이디어는 두 가지다. ① 진실 공급원의 위계를 문서로 고정하고, ② 사람의 규율에 기대던 것을 hook·CI·권한 설정으로 물리적으로 강제한다. 에이전트가 "하지 말아야 할 일"은 프롬프트로 부탁하는 게 아니라 도구 수준에서 차단된다.

진실 공급원 위계 (충돌 시 위가 이김)

1specs/requirements.mdEARS 형식 요구사항. 여기 없는 기능은 만들지 않는다 — 스코프 크리프의 방어선
2docs/adr/아키텍처 결정 기록 6건. 모순 구현 금지, 변경은 새 ADR 먼저
3CLAUDE.md최상위 규칙의 포인터 인덱스. 상세는 참조 파일로 위임해 항상 짧게 유지
4docs/design/DESIGN.mdUI 진실 공급원. 기능을 추가할 수 없어 충돌 시 requirements가 이긴다

3-1. 스펙 우선 · Deep Interview

구현 전에 요구사항을 EARS 형식("~하면, 시스템은 ~해야 한다")으로 작성하고, 모호한 항목(🟡)은 Deep Interview(질문 은행 기반의 구조화된 문답)로 전부 해소한 뒤 v1으로 동결했다. 🟡 8건 → 0건 → 동결 (2026-07-17, PR #7).

  • 모든 RQ에 ID가 있고, 그 ID가 브랜치명(feat/RQ-XX-*)· 테스트 파일명(rq-02-message-isolation.test.ts)·골den 케이스(GA-*)까지 관통한다. 요구사항 → 테스트 → PR의 추적성이 이름 규칙만으로 확보된다.
  • 비요구사항 명시: DM, 파일 전송, 계정, 영속 DB 등 "안 만드는 것"을 스펙에 박아두어, 에이전트의 선의의 기능 추가를 판정 가능하게 만들었다.
  • 동결 후 개정 절차: RQ-18(안 읽음 배지)은 디자인 단계에서 필요가 드러나자 v1.1 개정으로 추가했다. "구현 게이트"(해당 RQ의 코드 착수 시점) 이전이면 스펙만 개정 가능, 이후면 코드와 같은 PR에 동행해야 한다 — 드리프트의 조작적 정의.

참고: 같은 인터뷰 기법은 디자인에도 적용됐다 (디자인 인터뷰 D1~D16 → DESIGN.md 확보, Phase 3.5). 이 문서는 BE 관점이라 상세는 생략한다.

왜 의미가 있나
LLM 에이전트의 대표적 실패 모드가 "그럴듯한 기능을 알아서 추가"하는 스코프 크리프다. 스펙이 유일한 허가 목록이 되면 리뷰어(사람이든 에이전트든)가 diff를 "스펙에 있나?" 하나로 판정할 수 있다. 실제로 reviewer 에이전트의 첫 번째 검사 항목이 이것이다.

3-2. ADR — 아키텍처 결정 기록

아키텍처 갈림길 6개를 ADR로 기록하고, ADR과 모순되는 코드 변경을 금지했다 (변경하려면 새 ADR로 대체가 먼저).

번호갈림길결정
ADR-0001실시간 전송Node.js+TS + Socket.IO (FE: React+Vite), 단일 프로세스
ADR-0002메시지 영속성인메모리 링버퍼 (room당 50개), 재시작 소실 허용
ADR-0003사용자 식별닉네임 + 서버 발급 토큰, 퇴장 유예 30초, 활성 room 정의
ADR-0004global 채널예약된 상설 room (생성·삭제·leave 예외 명문화)
ADR-0005테스트 전략통합 중심 TDD, Vitest, 전송 계층만 대역 허용
ADR-0006배포Docker 단일 컨테이너, 스모크 = 골든 케이스 승격

특기할 점은 ADR이 테스트 전략과 구현 범위 판단의 근거로 실제 소비됐다는 것이다. 예: RQ-14(메시지 순서 보장)는 ADR-0001의 "단일 프로세스 이벤트 루프"가 순서를 자연 보장하므로, 테스트 4건이 구현 코드 0줄로 즉시 Green이 됐다 — 아키텍처 결정의 부수 효과를 테스트가 증명한 사례.

3-3. 진행 원장 (docs/progress.md)

모든 작업은 시작 전 원장에서 요구사항·참조 파일을 확인하고 상태를 🔄로 바꾼 뒤 진행하며, 완료 시 ✅ + 산출물 PR 번호를 기록한다. 원장에 없는 작업은 행을 추가한 뒤에만 시작할 수 있다.

  • 세션이 바뀌어도(컨텍스트가 초기화돼도) 에이전트가 "지금 어디까지 왔고 다음이 뭔지"를 원장 한 파일로 복원한다 — LLM의 세션 휘발성에 대한 대응.
  • RQ별 행에 리뷰에서 나온 후속 과제(⚠️)까지 병기해, 머지 후 잊히는 minor 지적이 없도록 했다. 6장 "남은 과제"가 이 기록에서 나온다.
  • 하네스 자체의 변경도 별도 원장(docs/harness/changelog.md)으로 추적한다 — 워크플로우의 진화·퇴행을 리뷰 가능하게.

3-4. TDD 3-세션 파이프라인 (tdd-workflow 스킬)

RQ 1건을 서로 격리된 에이전트 세션 3개가 릴레이로 구현한다. 데이터는 _workspace/{RQ-ID}/ 파일로만 전달하고, 중간 산출물은 감사 추적을 위해 보존한다.

단계에이전트모델역할과 제약
Redtest-writerSonnetRQ의 EARS 문장 + 골든 케이스만 보고 실패 테스트 작성. 구현 코드 작성 금지. 테스트 커밋을 구현보다 선행시킨다
GreencoderSonnet실패 테스트를 통과시키는 최소 구현. 테스트 파일 수정 금지 — 모순 발견 시 수정이 아니라 보고
평가evaluatorOpus구현 세션의 대화·의도 설명 없이 파일과 코드만 보고 전체 테스트 재실행·골든 커버리지 대조·테스트 약화 감지 → PASS/FAIL

왜 세션을 쪼개는가 — 격리가 곧 가치

  • 테스트 작성자가 구현을 보면 구현에 맞춘 테스트가 된다 → Red를 분리.
  • 평가자가 구현 세션의 맥락(작성자의 자기 설명)을 공유하면 자기 채점이 된다 → 평가를 분리하고, 프롬프트에 RQ-ID와 폴더 경로만 전달.
  • 에이전트 간 자유 통신(팀 모드)은 이 격리를 깨므로 의도적으로 쓰지 않았다 — 구조가 곧 정책이다.
  • 판정 모델 상향: 생성(Sonnet)보다 판정(evaluator·reviewer = Opus)에 더 강한 모델을 배치했다. 파이프라인의 신뢰도 상한은 판정 품질이 정한다는 판단.
  • 실패 시 프로토콜: FAIL이면 보고서를 입력으로 coder 1회 재호출, 2연속 FAIL이면 자동 반복을 멈추고 사람에게 보고. 테스트 약화로 우회하는 것은 evaluator의 검사 항목이자 금지 규칙.

3-5. 독립 리뷰 게이트 (review-gate 스킬)

1인 프로젝트에서는 사람 리뷰어가 없고, GitHub은 자기 PR을 자기가 승인할 수 없다. 그래서 격리 세션의 reviewer 에이전트(Opus) APPROVE를 머지의 필요조건으로 삼았다. CI(결정론적 검사)와 reviewer(추론적 검사)의 이중 게이트이며, main 브랜치 보호(status check 필수)가 물리적으로 받친다.

  • reviewer에게 diff·관련 RQ/ADR 목록만 주고 구현 세션의 의도 설명은 주지 않는다 — 작성자 논리에 오염되지 않은 리뷰가 게이트의 존재 이유.
  • 검사 관점: 스펙 근거 없는 변경(스코프 크리프), ADR·CLAUDE.md 위반, 보안, 과잉 설계. REQUEST_CHANGES 2연속이면 자동 반복을 멈추고 설계 재검토 신호로 취급.
  • 실효성 증거: RQ-05 리뷰에서 잘못된 URL 인코딩에 서버가 크래시하는 결함 (B-1, 무인증 원격 DoS)을 blocker로 적발해 머지 전에 수정됐고, 스펙 개정 PR #11 리뷰는 "활성 room 미정의"(major)를 잡아 test-writer가 멈출 상태를 사전에 해소했다.

3-6. 가드레일 — hook·CI·권한으로 강제

센서 카탈로그(harness/sensor-catalog.md)는 가드레일을 Guide(행동 전 읽는 규칙) / Sensor(행동 후 관찰·교정), 결정론적(Comp) / 추론적(Inf)으로 분류한 한 장의 지도다. 원칙: "반드시"는 hook·CI로 강제하고, "권장"만 문서(Guide)로 남긴다.

가드레일배치 시점강제 수단
스펙 동결 게이트 — 🟡 미해소 시 구현 차단구현 파일 수정 직전 + PRPreToolUse hook exit 2 + CI fail
골든 정답 수정 승인 — 에이전트가 자기 정답을 쓰지 못하게evals/golden/** 쓰기 시permissions ask 게이트
파일 수정 후 빠른 검사 (lint+typecheck, 실측 1.4초)수정 직후PostToolUse hook → check.sh --fast
전체 테스트 + 테스트-코드 동행 검사CI, PR 머지 게이트ci.yml → vitest run (테스트 ~2초, check.sh 전체 3.2초)
독립 평가·독립 리뷰구현 직후 / 머지 전오케스트레이터 스킬 + 브랜치 보호
배포 후 스모크 (골든 GA-01·GA-04 프로덕션 승격)main 머지 → 배포 직후deploy.yml → smoke.sh
트래젝토리 로그 (세션 행동 기록)세션 종료Stop hook

운영 규칙 (센서 카탈로그에서)

  • 왼쪽 배치 원칙: 센서는 가능한 한 이른 시점(수정 직후 > pre-commit > CI > 리뷰)에 둔다. 피드백 지연이 짧을수록 에이전트의 자기 교정이 싸다.
  • 에러 메시지에 고치는 법을 담는다 — 읽는 주체가 사람이 아니라 에이전트이기 때문이다.
  • 같은 실수 2회 → 센서 또는 Guide 하나 추가 (둘 다는 과잉). 실제 운영에선 반복을 기다리지 않고 1회 사고 만에 조치한 사례도 있다: RQ-01 파이프라인에서 Red 증거가 vitest 출력만 캡처해 타입 버그를 놓치자, test-writer 정의에 tsc --noEmit 증거를 의무화했다 (2026-07-20 changelog).

3-7. 골든 eval 세트 · 메트릭

두 트랙의 골든 케이스

  • 트랙 A — 제품 행동 (27케이스): "사용자가 X 하면 Y여야 한다"를 JSONL로 정의하고, 각 케이스를 통합 테스트로 구현해 verify 필드로 연결한다. 이 중 GA-01·GA-04는 배포 스모크로 프로덕션 승격됐다 — 골든 케이스가 문서 → 테스트 → 배포 검증까지 재사용되는 구조.
  • 트랙 B — 하네스 행동 (5케이스): "coder가 테스트를 약화시키면 evaluator가 적발해야 한다" 같은 워크플로우 자체의 행동을 rubric으로 정의하고 사람이 체크한다. 하네스를 바꿀 때의 회귀 테스트다. 단, 5건 모두 아직 정식 실행 기록이 없는 todo 상태 — 절차 정의까지가 현재 지점이다 (6장 참조).
  • 정답은 사람이 쓴다: 에이전트가 자기 정답을 쓰는 순환을 permissions(ask 게이트)로 차단. 승격 루프 — 트래젝토리 로그에서 이상했던 세션의 입력을 새 케이스로 추가한다. 케이스 수 13 → 32로 성장.

메트릭 베이스라인 (M1~M7)

"하네스가 먹혔는가"를 감이 아니라 숫자로 회고하기 위한 지표: 스펙 밖 변경 수(M1), 스펙-코드 동행률(M2), 테스트 선행률(M3), time-to-merge(M4), 재작업률(M5), 세션당 도구 호출(M6), 골든 케이스 수(M7). 전부 git·CI·로그 등 기존 산출물에서 수집하며 별도 인프라를 만들지 않았다. M4~M6은 "하네스가 속도를 죽이고 있지 않은지"를 감시하는 부작용 지표라는 점이 특징.

4. 무엇이 좋았나 — 의미와 증거

아래는 워크플로우가 실제로 사고를 잡았거나 비용을 줄인 기록이다. 전부 저장소의 원장·changelog·PR에 남아 있다.

적발 골든 케이스가 격리 구멍을 잡다 — GA-10
RQ-02(메시지 격리) 구현에서 "발신자가 room 참여자인지" 검증이 빠진 이월 구멍을 골든 케이스 GA-10이 잡아, socket.rooms.has 검증이 추가됐다 (PR #16). 보안 성격의 결함이 스펙 단계에서 만든 케이스로 걸러진 사례.
적발 독립 리뷰가 크래시를 머지 전에 잡다
RQ-05(배포) 리뷰에서 reviewer가 잘못된 URL 인코딩에 서버가 크래시하는 결함(B-1, 인증 없이 원격에서 트리거 가능한 DoS)을 blocker로 판정, 수정 후 머지됐다. 구현·평가 세션이 모두 놓친 것을 세 번째 시각이 잡았다 — 격리된 다중 검증 계층의 존재 이유.
적발 스펙 리뷰가 파이프라인 정지를 예방하다 — PR #11
RQ-18 스펙 개정 리뷰에서 "활성 room 미정의"(major 6건 중 하나)가 적발됐다. 그대로 진행했다면 test-writer가 테스트를 쓸 수 없어 멈출 상태였다. 스펙도 코드처럼 리뷰 게이트를 통과시킨 것이 유효했다.
학습 하네스가 스스로 진화하다 — 사고 → 센서 추가 루프
Red 증거가 vitest 출력만 캡처해 테스트 파일의 타입 버그가 coder 단계에서야 발견되자, test-writer 정의와 ADR-0005에 tsc --noEmit 증거를 의무화했다. 사고 → 센서 추가 → changelog 기록의 학습 루프가 실제로 작동한 사례.
효율 아키텍처 결정이 구현을 대체하다 — RQ-14
메시지 순서 보장은 ADR-0001(단일 프로세스)의 부수 효과라서 테스트 4건이 구현 0줄로 즉시 Green. evaluator가 11회 반복 실행으로 가드 강도를 검증했다. "요구사항 = 코드 추가"라는 등식을 ADR이 끊어준 사례.
신뢰 완료 주장에는 항상 실행 증거
모든 RQ의 완료 보고에 테스트 실행 출력이 첨부됐고, evaluator가 별도 세션에서 재실행으로 교차 확인했다 (예: RQ-18은 44/44 재검증). "된 것 같다"가 아니라 "이 출력이 증거다"가 기본값이 됐다.

일반화하면

  • 에이전트 신뢰성 문제를 구조로 해결: 자기 채점, 구현에 맞춘 테스트, 테스트 약화, 스코프 크리프 — LLM 에이전트의 전형적 실패 모드마다 대응하는 격리 또는 게이트가 있다. 프롬프트 지시("하지 마")가 아니라 구조(할 수 없음)로 막는다.
  • 1인 개발에 다역 검증 체계: 작성자·평가자·리뷰어를 격리 세션으로 분리해, 솔로 프로젝트에서도 상호 검증이 있는 팀의 형태를 재현했다.
  • 추적성: RQ-ID 하나로 스펙 문장 → 골든 케이스 → 테스트 파일 → 브랜치 → PR → 원장 행이 전부 연결된다. 감사·회고 비용이 낮다.
  • 속도를 해치지 않았다: 게이트가 겹겹인데도 마감 10일 전에 전 RQ가 끝났다. 빠른 피드백(수정 직후 1.4초 검사, check.sh 전체 3.2초)을 왼쪽에 배치한 설계 덕이 크다.

7. 보완 제안 작성자(Claude) 관점

운영하며 드러난 구조적 공백과 트레이드오프에 대한 제안이다. 우선순위 순.

① E2E 브라우저 스모크 추가 — 검증 사각지대 해소
현 테스트 44개 중 42개가 서버 통합 테스트이고, 클라이언트 테스트는 입장 화면 (EntryScreen) 렌더 1건뿐이다. 핵심 클라 배선(localStorage 토큰, 배지 렌더, 자동 스크롤)은 추론적 리뷰로만 검증됐고, 실제로 UI 결함(메시지 겹침·스크롤)은 수동 테스트에서야 발견돼 PR #29로 수정됐다. Playwright로 골든 케이스 2~3개 (입장→메시지→배지)만 E2E로 승격하면 이 사각지대가 닫힌다. 전체 E2E화는 과잉 — 스모크 수준이면 충분하다.
② RQ-16(동시 100명) 실측 증거 확보
동시 접속 100명은 ADR-0001의 제약 조건으로만 반영됐고 측정된 적이 없다. "완료 주장에는 실행 증거"라는 이 프로젝트의 자체 원칙에 비추면 유일하게 증거가 없는 요구사항이다. socket.io-client로 100 연결 × 메시지 브로드캐스트를 도는 스크립트 하나면 스모크에 편입할 수 있다.
③ 메트릭 수집 스크립트화
M2(스펙-코드 동행률)·M3(테스트 선행률)·M7(골든 수)은 git log와 wc로 결정론적으로 계산 가능하다. 주간 수동 갱신은 회고가 밀리면 같이 밀린다 — 7/31 회고 전에 수집 스크립트를 만들면 회고 비용이 내려가고, 이후 프로젝트에서 재사용된다.
④ 파이프라인 비용 대비 경량 트랙
RQ 1건당 격리 세션 3개 + 리뷰 1개(Opus 2회)는 품질 근거가 확실하지만 비용도 실재한다. RQ-14처럼 구현 0줄로 끝난 건에도 전체 파이프라인이 돌았다. diff 규모나 위험도(서버 상태 변경 유무)에 따라 evaluator를 생략하고 review-gate만 거치는 경량 트랙을 정의하면 비용이 내려간다. 단, 분기 기준은 에이전트 재량이 아니라 명문화된 규칙이어야 한다 — 재량에 맡기면 게이트 우회 통로가 된다.
⑤ 하네스 템플릿화 — 프로젝트 자산에서 팀 자산으로
원장·스펙 동결 게이트·3-세션 파이프라인·센서 카탈로그·골든 승격 루프는 채팅 앱과 무관한 범용 구조다. .claude/(agents·skills·hooks·settings)와 문서 뼈대를 도메인 무관 템플릿 저장소로 추출하면, 다음 프로젝트는 Deep Interview부터 시작할 수 있다. L&L의 실질 산출물로 이것을 제안한다.
⑥ 실서비스 전환 시 재결정 지점 명시
인메모리 상태(히스토리·세션·안읽음)는 스펙상 허용이지만, 단일 컨테이너 재배포 = 전체 상태 소실이므로 배포 시마다 사용자 세션이 끊긴다. 사내 실사용으로 간다면 ADR-0002(영속성)·ADR-0003(세션)의 재결정이 선행 조건임을 배포 문서에 명시해두는 것이 안전하다. 지금 고치자는 게 아니라, 미래의 논의가 ADR 절차를 타도록 입구를 적어두자는 제안이다.

8. 팀 적용 제언

전부를 한 번에 도입할 필요는 없다. 비용 대비 효과 순으로, 기존 팀 프로세스에 독립적으로 얹을 수 있는 것부터:

  1. 스펙 동결 게이트 하나만이라도 — "모호한 요구사항이면 구현 도구를 차단"은 hook 스크립트 1개로 이식되며, 에이전트 사용 여부와 무관하게 효과가 있다. 가장 싸고 가장 자주 발동한 가드레일이었다.
  2. 완료 주장 = 실행 증거 규칙 — PR 본문에 테스트 실행 출력 첨부를 관례화. 도구 없이 문화로 시작할 수 있다.
  3. 독립 리뷰 게이트 — 사람 리뷰어가 있는 팀이라도, 사람 리뷰 전에 에이전트 리뷰를 1차 필터로 두면 사람의 리뷰 비용이 내려간다. 핵심은 리뷰 세션에 작성 맥락을 주지 않는 격리다.
  4. 골든 케이스 승격 루프 — 장애·리뷰 지적을 골든 케이스로 승격하고 정답은 사람이 쓴다는 규칙. 기존 회고 프로세스에 붙이기 쉽다.
  5. 3-세션 TDD 파이프라인 — 도입 비용이 가장 크므로 마지막. 에이전트에게 구현을 위임하는 비중이 높아질수록 투자 가치가 커진다.
