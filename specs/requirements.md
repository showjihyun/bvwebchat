# 요구사항 — 웹 채팅 앱 (v0 · 초안)

> 형식: EARS 스타일. "~하면(WHEN), 시스템은 ~해야 한다(SHALL)."
> 상태: ✅확정 / 🟡PENDING(인터뷰로 확정) / 각 항목 ID는 테스트·브랜치명에 사용.

## 1. 확정된 요구사항 (회의 스펙 기반)

- **RQ-01** ✅ 사용자가 room에 참여하면, 시스템은 그 사용자를 해당 room의
  수신자 목록에 추가해야 한다.
- **RQ-02** ✅ 사용자가 room에 메시지를 보내면, 시스템은 해당 room 참여자
  전원에게만 그 메시지를 전달해야 한다. (다른 room·비참여자에게 절대 노출 금지)
- **RQ-03** ✅ 사용자가 room을 떠나면, 시스템은 이후 해당 room 메시지를
  그 사용자에게 전달하지 않아야 한다.
- **RQ-04** ✅ 사용자가 global 채널에 메시지를 보내면, 시스템은 접속 중인
  모든 사용자에게 전달해야 한다.
- **RQ-05** ✅ 마감: 7/31까지 위 기능이 동작하는 상태로 배포 가능해야 한다.

## 2. PENDING — Deep Interview로 확정할 것

- **RQ-10** 🟡 사용자 식별/인증: (닉네임만? 계정+로그인? 세션 유지?)
- **RQ-11** 🟡 메시지 영속성: 새로고침/재접속 시 이전 메시지 표시 여부와 범위
  (최근 N개? 전체? 없음?)
- **RQ-12** 🟡 room 생성/삭제 권한: 누가 room을 만들고 없앨 수 있는가
- **RQ-13** 🟡 room 목록/검색: 사용자가 room을 어떻게 발견하는가
- **RQ-14** 🟡 전달 보장 수준: 순서 보장? 유실 허용? 중복 처리?
- **RQ-15** 🟡 접속 상태 표시: 참여자 목록, 온라인 표시, 타이핑 표시 필요 여부
- **RQ-16** 🟡 동시접속 규모: 데모(~10명)인가 실사용(~수백)인가 → 아키텍처 결정에 직결
- **RQ-17** 🟡 배포 대상: 로컬 데모 / 사내망 / 공개 웹

## 3. 비요구사항 (명시적 스코프 제외)

- 파일/이미지 전송, DM(1:1), 알림 푸시, 모바일 앱 — v1에서 제외
  (필요해지면 스펙 개정 + ADR)
