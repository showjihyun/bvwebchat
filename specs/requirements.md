# 요구사항 — 웹 채팅 앱 (v1 · 스펙 동결)

> 형식: EARS 스타일. "~하면(WHEN), 시스템은 ~해야 한다(SHALL)."
> 상태: ✅확정 / 각 항목 ID는 테스트·브랜치명에 사용.
> 2026-07-17 Deep Interview로 전 항목 확정 (근거: specs/interview/answers.md).

## 1. 확정된 요구사항 (회의 스펙 기반)

- **RQ-01** ✅ 사용자가 room에 참여하면, 시스템은 그 사용자를 해당 room의
  수신자 목록에 추가해야 한다.
- **RQ-02** ✅ 사용자가 room에 메시지를 보내면, 시스템은 해당 room 참여자
  전원에게만 그 메시지를 전달해야 한다. (다른 room·비참여자에게 절대 노출 금지)
- **RQ-03** ✅ 사용자가 room을 떠나면, 시스템은 이후 해당 room 메시지를
  그 사용자에게 전달하지 않아야 한다.
- **RQ-04** ✅ 사용자가 global 채널에 메시지를 보내면, 시스템은 접속 중인
  모든 사용자에게 전달해야 한다. room에 참여 중인 사용자도 global 메시지를
  수신해야 한다. (인터뷰 질문 13)
- **RQ-05** ✅ 마감: 7/31까지 위 기능이 동작하는 상태로 배포 가능해야 한다.

## 2. 인터뷰로 확정된 요구사항 (2026-07-17)

- **RQ-10** ✅ 사용자가 접속하면, 시스템은 닉네임 입력만으로 사용자를 식별해야
  한다 (계정·비밀번호 없음). 이미 사용 중인 닉네임이 입력되면, 시스템은 자동
  접미사를 붙여 고유한 닉네임을 부여해야 한다. 같은 브라우저에서 새로고침하면,
  시스템은 동일 사용자로 인식하고 참여 중이던 room을 복원해야 한다
  (서버 프로세스가 유지되는 동안).
- **RQ-11** ✅ 사용자가 room에 입장하면, 시스템은 해당 room의 최근 50개
  메시지를 표시해야 한다. 메시지는 인메모리로 보관하며, 서버 재시작 시
  소실을 허용한다 (영속 DB 비요구).
- **RQ-12** ✅ 사용자가 새 이름으로 room 생성을 요청하면, 시스템은 권한 제한
  없이 room을 생성해야 한다. 마지막 참여자가 room을 떠나면, 시스템은 해당
  room을 자동 삭제해야 한다.
- **RQ-13** ✅ 시스템은 존재하는 모든 room의 목록을 모든 사용자에게 제공해야
  한다. 동일 이름의 room은 동시에 둘 이상 존재할 수 없다 (이름 = 고유 식별자).
- **RQ-14** ✅ 한 room에서 메시지들이 전송되면, 시스템은 해당 room 참여자
  전원에게 서버 도착 순서와 동일한 순서로 전달해야 한다. 일시 단절 후
  재접속 시 별도 정밀 복구는 하지 않으며, 재입장 히스토리(RQ-11)로 대체한다.
- **RQ-15** ✅ 시스템은 각 room의 현재 참여자 목록을 표시해야 한다.
- **RQ-16** ✅ 시스템은 동시 접속 100명까지 정상 동작해야 한다 (단일 서버 허용).
- **RQ-17** ✅ 시스템은 사내망/로컬 환경의 단일 서버로 배포 가능해야 한다.
  HTTPS는 선택 사항이다.

## 3. 비요구사항 (명시적 스코프 제외)

- 파일/이미지 전송, DM(1:1), 알림 푸시, 모바일 앱 — v1에서 제외
  (필요해지면 스펙 개정 + ADR)
- 계정/로그인·비밀번호 (RQ-10 인터뷰 결정)
- 메시지 영속 DB, 서버 재시작 후 복원 (RQ-11 인터뷰 결정)
- 온라인/오프라인 표시, 타이핑 표시 (RQ-15 인터뷰 결정)
- 비공개/초대제 room (RQ-13 인터뷰 결정)
