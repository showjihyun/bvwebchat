# Deep Interview 질문 뱅크

> 사용법: 새 세션(스펙 전용, 코드 금지)에서 Claude에게 이 파일로
> 인터뷰를 시켜 `specs/requirements.md`의 🟡PENDING을 확정한다.
> 답이 나올 때마다 해당 RQ를 EARS 문장으로 고쳐 쓰고 ✅로 바꾼다.
> F·G처럼 RQ가 아닌 섹션의 답은 하네스 문서(docs/adr/ 예약표,
> harness/, evals/)에 반영한다.

## A. 사용자와 인증 (→ RQ-10)

1. 사용자는 이름만 입력하면 되는가, 계정 생성/로그인이 필요한가?
2. 같은 이름으로 두 명이 접속하면? (거부 / 허용 / 자동 접미사)
3. 브라우저 새로고침 후에도 같은 사용자로 인식돼야 하는가?

## B. 메시지 수명 (→ RQ-11, RQ-14)

4. room에 새로 들어온 사람은 이전 대화를 보는가? 본다면 몇 개까지?
5. 서버 재시작 후 메시지가 남아 있어야 하는가? (= DB 필요 여부의 분기점)
6. 메시지 순서가 뒤바뀌면 문제가 되는 사용 시나리오가 있는가?
7. 잠깐 끊겼다 재접속하면 놓친 메시지를 받아야 하는가?

## C. room 수명주기 (→ RQ-12, RQ-13)

8. room은 누가 만드는가? (아무나 / 관리자만 / 사전 정의 고정)
9. 빈 room은 자동 삭제되는가?
10. room 목록은 전체 공개인가? 비공개/초대제 room이 필요한가?
11. room 이름 중복은 허용되는가?

## D. global 채널의 정체 (→ RQ-04 정밀화)

12. global은 "특수한 room"인가, 별개 개념인가? (UI와 모델 설계 분기점)
13. room 안에 있는 동안에도 global 메시지를 받는가?

## E. 규모와 운영 (→ RQ-16, RQ-17)

14. 동시 접속 예상 최대치는? (10 / 100 / 1000+)
15. 어디에 배포되는가? HTTPS 필요한가?
16. 이 앱의 수명은? (일회성 데모 / 계속 발전시킬 베이스)

## F. L&L 메타 질문 (하네스 자체)

17. 이 프로젝트에서 "성공한 워크플로우"의 판정 기준은 무엇인가?
    (예: 스펙 밖 변경 0건, 회고 때 재현 가능한 기록 존재)
18. 팀원이 몇 명이고, 누가 리뷰 게이트를 담당하는가?

## G. 테스트 전략과 TDD (→ ADR-0005, M3, GB-02/04/05)

> CLAUDE.md는 "테스트 먼저"를 선언만 한다. 아래 답이 없으면
> TDD는 구호로 끝난다 — 답은 ADR-0005와 scripts/check.sh에 반영한다.

19. 골든 케이스(GA-*)는 어느 레벨의 테스트로 구현하는가?
    (통합 테스트 중심 / 단위 병행 / 브라우저 E2E 포함 — `verify` 필드 확정)
20. "테스트 먼저"의 증거는 무엇으로 삼는가?
    (테스트 커밋이 구현 커밋보다 선행 / PR에 Red 실패 출력 첨부 / 둘 다)
    — M3(테스트 선행률) 측정 방법이 여기서 결정된다.
21. 실시간 전송(WebSocket 등)을 결정론적으로 테스트하기 위해
    무엇을 허용하는가? (인메모리 전송 대역, fake timer, 대기 시간 상한)
    — flaky 테스트는 머지 게이트의 신뢰를 죽인다.
22. 테스트 더블(mock/stub)은 어디까지 허용하는가?
    (전송 계층만 / 저장소 포함 / 금지 — 실제 인스턴스 사용)
23. 속도 예산: `check.sh --fast`(수정 직후 hook)와 CI 게이트의
    시간 상한은? (예: fast 5초 / CI 3분) 초과하면 어떤 테스트를
    게이트 밖(야간·수동)으로 빼는가?
24. TDD 의무의 예외는 무엇인가? (문서·설정·스파이크 탐색 코드 등)
    스파이크 코드는 어떻게 처리하는가? (버리는 브랜치 / 테스트 소급 후 편입)
25. 커버리지 수치 목표를 둘 것인가, M3(테스트 선행률 ≥80%)만으로
    충분한가? (커버리지는 게이트가 되는 순간 게임의 대상이 된다)

## H. 디자인 스타일과 UI 산출물 (→ docs/design/DESIGN.md)

> 이 섹션은 인터뷰에서 **반드시 짚고 넘어간다** — 답 없이 UI 구현을
> 시작하면 화면마다 스타일이 표류한다. AskUserQuestion으로 물을 때
> 26번의 권장안을 첫 번째 옵션 "(Recommended)"로 제시하라.

26. Design Style과 디자인 관련 md는 어떻게 확보하는가?
    - **(권장) 클로드 디자인으로 먼저 작업**: 클로드 디자인으로
      Design Style·디자인 md를 먼저 생성하고, 그 산출물을
      `docs/design/DESIGN.md`로 커밋 → 이후 모든 UI 작업은
      이 파일을 이어받아 진행 (디자인의 진실 공급원)
    - 대안: 수동 스타일 가이드 작성 / 기존 디자인 시스템 차용
27. DESIGN.md가 커밋되면 CLAUDE.md 진실 공급원 목록에 추가하는가?
    (권장: 추가 — UI 구현이 DESIGN.md와 모순되면 드리프트로 취급)
28. 구현 중 디자인 변경이 필요하면 절차는?
    (권장: DESIGN.md 개정을 코드 변경과 같은 PR에 — 스펙 변경 규칙과 동일)

### H-2. 디자인 방향 (Q26에서 "클로드 디자인 선행"을 택했다면 필수)

> **교훈(2026-07-17)**: Q26~28은 디자인을 *어떻게 확보할지*만 묻는다.
> *어때야 하는지*를 묻지 않으면 hand-off 문서가 요구사항만 담게 되고,
> 디자이너는 무난하지만 몰개성한 결과를 낸다. 아래를 반드시 함께 물어라.
> 실제 진행 결과: `specs/interview/design-answers.md`

29. 어느 제품과 비슷한 느낌인가? (가장 강한 디자인 신호 — 참조점)
30. 앱의 성격·톤은? (절제된 업무 도구 / 친근함 / 기술적)
31. 브랜드 제약이 있는가? (사내 컬러·로고·앱 이름)
32. 화면 밀도는? (컴팩트 / 편안한 여백)
33. 컬러 방향은? 다크 모드가 필요한가? (산출물 범위·구현 비용에 직결)
34. 도메인 고유 표현 방식은? (이 앱: 메시지 버블 vs 플랫, 아바타 유무)
35. UI 언어와 폰트는? (한국어면 줄높이·자간 명시 필요. 사내망이면 웹폰트
    CDN 접근 가능 여부를 확인하라)
36. 모션 정도와 접근성 기준은? (WCAG AA는 팔레트 설계 시점에 반영해야 저렴)
37. 도메인 고유 인터랙션은? (이 앱: 안 읽음 표시 방식, 알림 표현)
    — **주의**: 표시 방식이 무엇이든("굵게+점"이든 "숫자 배지"든) **"안 읽음"은
    상태 추적을 요구한다** — 어디까지 읽었는지를 어딘가 보관해야 하고, 그 상태는
    스펙 항목이다. 개수를 안 세면 가벼워질 뿐 공짜가 되지 않는다. 스펙에 없으면
    개정이 먼저다.
    → 실제 사례: 이 프로젝트는 "굵게+점"을 "카운트를 안 세니 스펙 확장 없음"으로
      오판했다가, 리뷰에서 상태 자체가 확장임을 지적받고 RQ-18을 신설했다
      (`design-answers.md` D13 → D13-r). 선을 **개수**가 아니라 **상태**에 그어라.
38. 진입 화면과 보조 패널의 기본 상태는? (랜딩 페이지 vs 최소 폼 /
    상시 노출 vs 토글) — 스펙이 "표시해야 한다"만 요구할 때 노출 방식은
    디자인 재량이다.

## 인터뷰 종료 조건

- requirements.md에 🟡가 0개
- 답변이 서로 모순되지 않음 (예: "영속성 불필요"인데 "재접속 시 복구 필요")
- 아키텍처 갈림길 5개(전송/영속성/인증/상태/테스트 전략)에 대한
  입력이 모두 확보됨
- 디자인 스타일 확보 방식(H)과 **디자인 방향(H-2)**이 모두 결정됨 — 둘 중
  하나라도 미결이면 UI 구현 착수 금지. 방향 없이 확보 방식만 정하면 hand-off
  문서가 요구사항만 담게 되어 몰개성한 산출물이 나온다 (2026-07-17 실제 사례)
